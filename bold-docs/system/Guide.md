# Documentation Guide

This is the living orientation document for `.documentation/`. It reflects the **current** state of the project's documentation surface.

---

## Directory Map

Migrated off DevSpark onto [Bold](https://github.com/MakeBoldSolutions) on 2026-07-09. Current layout:

```
bold-docs/
├── backbone.md                     ← Project constitution (non-negotiable principles), migrated
├── project.json                    ← Project genome — discovered stack facts
├── system/                         ← This file plus ARCHITECTURE.md, API_DEPENDENCIES.md,
│                                      CACHE_BUSTING.md, CHANGELOG.md, CONTRIBUTING.md,
│                                      DEPLOYMENT.md, PROMPTSPARK_CHAT.md, SECURITY.md
│   └── releases/                   ← Versioned release archives (see Release Archives below)
├── overrides/                      ← Carried-forward team script overrides (flagged for review —
│                                      DevSpark's command surface differs from Bold's)
└── features/                       ← Active feature specs (Bold tier-gated planning)

.archive/                           ← Repo root. Completed specs, repo-story, historical decisions.
                                       Human-readable only — never read by an agent.
```

## How to Use Bold Commands

Bold commands are invoked via `/bold-plan`, `/bold-build`, `/bold-ship` (and their subcommands) in any adapter host — Claude Code, Codex, Copilot. Bold's three-command surface doesn't map one-to-one onto DevSpark's; some DevSpark commands fold into a broader Bold command, and a few have no Bold equivalent yet.

| DevSpark command | Bold equivalent |
|---|---|
| `/devspark.specify`, `/devspark.plan`, `/devspark.tasks` | `bold.plan` — tiered spec (Patch/Quick/Feature) covers requirements through task breakdown in one command |
| `/devspark.implement` | `bold.build` |
| `/devspark.harvest`, `/devspark.archive` | `bold.ship harvest` — classifies durable vs. historical and archives in one pass |
| `/devspark.pr-review` | `bold.ship review` |
| `/devspark.quickfix` | Not a separate command — the Patch tier within `bold.plan` |
| `/devspark.site-audit` | *(no Bold equivalent yet)* |
| `/devspark.release` | *(no Bold equivalent yet — see Release Automation below)* |

### Release Automation

Bold has no dedicated release command yet. The tag-driven GitHub Release flow (`.github/workflows/github-release.yml`) still works — it reads from `bold-docs/system/releases/vX.Y.Z/` — but the release notes themselves must currently be written by hand (or drafted by asking an agent directly) into that folder before tagging, rather than generated by a `/devspark.release`-style command.

## Constitution Location

**`bold-docs/backbone.md`** — The project constitution defines non-negotiable principles covering type safety, code quality, testing, documentation standards, component architecture, error handling, logging, security, styling, and deployment. All code reviews must verify compliance.

---

## Release Archives

Versioned release artifacts live under `bold-docs/system/releases/vX.Y.Z/`. Each release folder contains release notes, metrics, and archived feature specs for that release cut.
